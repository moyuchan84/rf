 다음 기능을 개발하기 위한 명세서(GEMINI.md)  를 잘 작성해줘.

1. 목표
반도체회사내 excel(nasca 걸려있는 문서)로 관리하고 있는 데이터를 database화 및 rag service 연계를 하기 위한 작업을 하려함
이때 excel 에 있는 데이터는 사람이 관리하다 보니 어느정도 규격은 있지만, 완전히 획일화되지 않은 만큼 비정형화된 데이터 형태(json 등)
로 저장이 필요한데, 이때 중요한 점은 원래 데이터를 엑셀로부터 읽어서 저장했지만 다시 원래대로 복원(셀서식까지포함)할 수 있어야함
거기에 더불어 rag service 및 웹으로 데이터 조회등을 연계할 것이기에 데이터에 대한 정제가 필요한 상황

2. 데이터 
반도체 photo 공정에 있어 align/overlay 등의 key에 대한 저장
데이터는 크게 processplan - beol_option - product 라는 계층적 구조를 갖고 있고 product 하위에 각각 photo key를 저장하는 구조
photo key의 경우 크게 기준정보성 테이블과 실제 photo reticle 의 overlay/align 등을 보정할때 사용하는 key data 가 있지만,
데이터를 json 등 비정형화된 데이터로 저장하면 좋을것 같기에, 컬럼으로 카테고리등을 구분하되 실제 데이터는 json 형태로 저장하면 좋을거 같음

1) 각 데이터별 타입
 - processplan : design_rule(str)
 -  beol_option : option_name(str)
 -  product : partid(str) product_name(str)
 -  photo_key : rfg_category(str- enum (common, option) ), photo_category(str - enum(info,key)), is_reference(bool), table_name(str), rev_no(int),  workbook_data(json), fillename(str),updater(str),update_date(datetime)
 ** 중요사항 : workbook_data(json) - workbook 에 대한 meta정보 + worksheets 에 대한 meta 정보 및 table data, sheet name ,column name 정보등 저장 필요
 ** worksheets 에 주요사항 
 - history sheet 있음,  data sheet 있음 -> history 인지 data 인지 구분 필요
 - data sheet : 단순히 테이블형태 데이터와 메타정보 + 테이블형태 2가지 형태로 구분됨
 -> 메타정보 + 테이블형태의 경우 열 중간(예를들어 D열)에 한 4-5줄정도 메타정보가 있고(D1,D2,D3,D4) 그리고 그아래줄부터는 테이블데이터가 있는 경우
     이때 메타정보 바로 아래 컬럼의 경우 컬럼헤더에 ; 가 포함되어있음. 그리고 테이블형태 데이터는 모두 첫번째 행은 컬럼이름임.
     하지만 사람이 저장하다보니 일괄된 이름으로 작성하지 않아, 중간중간 컬럼이름이 중복되거나, 누락된 경우가 있음.
     그에 따라 컬럼이름 : 셀값 등을 저장하기 위해서는 아마 컬럼이름에 대한 aliasing 이 필요하고, 그 aliasing 을 다시 원래 컬럼으로 복원할수있는 
     map<string,string> 형태 key:value 형태로 aliasing_column : original_column형태로 저장하고, 테이블데이터 저장할땐 aliasing_column:cell_value 형태로
    저장했다가 db-> excel 로 복원할때 다시 aliasing->original_column + cell value(with cell style)로 복원이 필요함. 즉 이 부분에서 데이터에 대한 parsing business logic (즉 meta 정보가 있다면 - photo_category 에서 보통 key, meta 정보가 없다면 photo_category 가 info 임
 -> 보통 worksheets 중에서 1번째 worksheet는 history, 2번째부터는 data sheet 임
 -> db 에 parsing 해서 저장했던 데이터를 다시 그대로 복원할 수 있어야함. 
 -> history 외 data sheets 는 web 과 rag 에서 데이터를 활용할 것이기에 구조적으로 구성해야함(meta info 가 중간열(생뚱맞게 C열/ E열 /F 열등 시작하기에 그 시작점에 대한 정보도 저장이 필요할것으로 보임)
 - > history sheet, data sheet 모두 1행 1열에 시작한다는 보장은 없음

3. 기술스택
 - vsto + webview2 (유지보수 관리하기 좋고 디자인 좋은 라이브러리로 추천)
 - db : postgresql 
 - backend : fastapi 
 - 구조 : clean architecture

4. VSTO 에 대한 요구사항
 - photo key table 에 대한 crud는 권한을 가진 사람만 진행할 수 있다. (리본메뉴에 대한 기능 권한 부여)
 - 특정 폴더에 있는 엑셀 파일들의 목록을 가져올 수 있다
 - 엑셀 workbook로부터 history / data sheet 에 대한 데이터를 parsing 하고, 3단계 트리구조로 사용자는 데이터(엑셀파일-워크시트-데이터내용) 확인할수있다.
 - workbook에서 parsing 한 worksheets 데이터에서 사용자가 history sheet 인지 data sheet 인지 선택할 수 있다.
 - 각 workbook 에 대해서 사용자가 rfg_category. photo_category, table_name, rev_no, is_reference에 대해서 선택할수 있다.
 -parsing 및 사용자가 특정속성에 대한 정보를 선택하고, workbooks 중 원하는 workbooks 을 선택한후 db에 저장할 수 있다.(processplan-beol_option-product 에 종속)
 - db에 저장된 product별  photo_key 목록(table 별 rev no가 여러개 있음) 을 불러올 수 있다.
 - db에 저장된 product 별 photo_key table 을 선택시 데이터(워크북 - 워크시트) 에 대한 데이터를 확인할 수 있다.
 - db에 product 별 photo_key table 목록을 복수개 선택해서 엑셀로 복원할 수 있다.
